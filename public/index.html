Requirements

  • Node.js 20.6.0+ (required for native .env file support)
  • OpenWeather API key (free tier: 1,000 calls/day)

Quickstart

1. Generate an OpenWeather API Key

Sign up for OpenWeather's One Call API v3.0 and generate your API key.

2. Clone the Repository 

git clone https://github.com/scottscharl/weather-api-v2.git
cd weather-api-v2

3. Get Your Coordinates

Option 1: Use LatLong.net - search for your city and copy the coordinates

Option 2: Right-click on a place in Google Maps and click the coordinates to copy

4. Configure Your Environment

Create a .env file in the project root: 

# Required
LATITUDE=40.7128
LONGITUDE=-74.0060
LOCATION_LABEL="New York City"
OPENWEATHER_KEY=your_openweather_key_here

# Auto-generated on first run if not provided
# API_KEY=

# Optional
PORT=4000
NODE_ENV=development

# CORS - Comma-separated allowed origins (optional)
# Development: localhost auto-allowed | Production: all origins (*) if not set
# ALLOWED_ORIGINS=https://yourdomain.com,https://app.yourdomain.com

Replace the values with your actual coordinates, location name, and OpenWeather API key.

5. Install Dependencies and Start 

npm install
npm start

If an API key wasn't found in your .env file, the app will generate one and display it. Copy the generated key, add it to your .env file, then restart the server.

6. Deploy

Deploy your server to a VPS or cloud platform and start building notifications, automations, and integrations with your apps and websites.

Security: Never commit your .env file. Use environment variables on your hosting platform. Keep your API_KEY secret and rotate it if compromised by generating a new one.

Cache Behavior

  • Default location: Automatically cached and refreshed every 5 minutes via cron job
  • Custom lat/lon requests: Bypass cache and make fresh API calls to OpenWeather
  • Rate limit protection: Caching keeps you under OpenWeather's 1,000 calls/day free tier limit (288 cached calls/day for default location)

API Endpoints

GET /api/

Returns simplified weather data with essential fields only.

Fields Returned:

  • location - Location name
  • current.temp - Current temperature
  • current.feels_like - Feels like temperature
  • current.humidity - Humidity percentage
  • current.wind_speed - Wind speed
  • current.weather - Weather description
  • forecast - Simplified forecast array

Query Parameters (Optional):

  • lat - Latitude (-90 to 90)
  • lon - Longitude (-180 to 180)
  • location - Custom location label

Example Request:

curl -H "X-API-Key: your-api-key" \
  "http://your-server:4000/api/?lat=40.7128&lon=-74.0060&location=New%20York"

Example Response:

{
  "location": "New York City",
  "current": {
    "temp": 72,
    "feels_like": 70,
    "humidity": 65,
    "wind_speed": 8.5,
    "weather": "Clear"
  },
  "forecast": [...]
}

GET /api/full

Returns complete weather data including all available fields from OpenWeather API.

Fields Returned:

  • All simplified fields above
  • uvi, visibility, pressure, dew_point
  • sunrise/sunset times
  • Detailed hourly and daily forecasts
  • alerts (if any)
  • And more...

View complete field documentation on OpenWeather's docs .

Query Parameters (Optional):

  • lat - Latitude (-90 to 90)
  • lon - Longitude (-180 to 180)
  • location - Custom location label

Example Request:

curl -H "X-API-Key: your-api-key" \
  "http://your-server:4000/api/full"

Error Responses

  • 401 Unauthorized - Missing or invalid X-API-Key header
  • 400 Bad Request - Invalid latitude or longitude values (specific error message will indicate the exact issue)
  • 500 Internal Server Error - Server or OpenWeather API error

Authentication

All API endpoints require an API key. Include the X-API-Key header in your requests: 

curl -H "X-API-Key: your-api-key" http://your-server:4000/api/

Or in JavaScript:

fetch('http://your-server:4000/api/', {
  headers: {
    'X-API-Key': 'your-api-key'
  }
})
  .then(res => res.json())
  .then(data => console.log(data));

Security Best Practices

‚ö†Ô∏è CRITICAL SECURITY NOTES:

  • API_KEY is for server-to-server use only - Never expose it in client-side code, public repositories, or browser applications
  • Use HTTPS in production - Always deploy behind HTTPS to protect API keys in transit
  • Restrict CORS origins - Set ALLOWED_ORIGINS in production instead of allowing all origins (*)
  • Never commit .env files - Add .env to .gitignore and use environment variables on your hosting platform
  • Rotate compromised keys - If your API_KEY is exposed, generate a new one immediately

CORS Configuration

Control which domains can access your API: 

# In .env file
ALLOWED_ORIGINS=https://yourdomain.com,https://app.yourdomain.com
  • Development: Localhost origins are automatically allowed
  • Production without ALLOWED_ORIGINS: All origins allowed (*) - not recommended
  • Production with ALLOWED_ORIGINS: Only specified domains allowed - recommended

Monitoring & Health Checks

GET /health

Health check endpoint (no authentication required). Returns server status and cache information.

Example Request:

curl http://your-server:4000/health

Example Response:

{
  "status": "ok",
  "timestamp": "2025-10-26T12:34:56.789Z",
  "uptime": 3600,
  "lastCacheUpdate": "2025-10-26T12:30:00.000Z",
  "cacheAge": 296,
  "error": null
}

Use this endpoint for uptime monitoring and debugging cache issues. Returns 503 status if there are cache errors.

Cache File Location

Weather data is cached in: ./data/weather_cache.json

Cache is automatically updated every 5 minutes. Check /health endpoint to see last successful update time and any errors.
